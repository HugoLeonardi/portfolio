Déploiement Cloud Nextcloud sur AWS

Infrastructure as Code (IaC) avec Terraform — Mission E6 BTS SIO SISR

Mission E6 — Déploiement Cloud (AWS)

🎯 Contexte et Problématique

Situation initiale

Dans le cadre de ma mission E6, j'ai répondu à un besoin de stockage collaboratif sécurisé. L'objectif était de proposer une solution Cloud fiable, sans dépendre d'un serveur physique sur site, et dont le déploiement serait entièrement automatisé et reproductible.

Problématique

Comment déployer une infrastructure Nextcloud complète sur AWS, de façon reproductible et sécurisée, sans aucune intervention manuelle après le lancement ?

Objectifs

1

Infrastructure as Code

Provisionner l'ensemble des ressources AWS (VPC, EC2, RDS, S3) via Terraform.

2

Architecture trois tiers

Séparer les couches web, base de données et stockage pour la sécurité et la scalabilité.

3

Sécurité Cloud

Appliquer le principe de moindre privilège via Security Groups et rôles IAM.

🏗️ Architecture technique

L'infrastructure repose sur trois couches distinctes déployées dans un VPC dédié (10.0.0.0/16), isolant chaque composant du reste d'Internet.

🌐

Couche Web — EC2 (sous-réseau public)

Instance t3.small sous Amazon Linux 2023. Apache + PHP 8.4 + Nextcloud. Configurée automatiquement via un script User Data au premier démarrage.

🗄️

Couche Base de données — RDS MySQL (sous-réseau privé)

Instance RDS managée par AWS, inaccessible depuis Internet. Placée dans 2 zones de disponibilité distinctes (exigence AWS RDS).

📦

Couche Stockage — S3

Bucket S3 utilisé comme stockage objet de Nextcloud. L'EC2 y accède via un rôle IAM — sans aucune clé en dur dans le code.

⚙️ Réalisation

Phase 1 Infrastructure

Codage Terraform

  • Définition du VPC, des sous-réseaux publics et privés.
  • Configuration des Security Groups (filtrage strict par port et source).
  • Déclaration du rôle IAM pour l'accès S3 depuis l'EC2.
  • Provisionnement RDS avec DB Subnet Group multi-AZ.
Phase 2 Scripting

Automatisation Bash (User Data)

  • Installation de la stack Apache + PHP 8.4 + extensions Nextcloud.
  • Téléchargement et déploiement automatique de Nextcloud.
  • Configuration d'Apache via VirtualHost dédié.
  • Initialisation de Nextcloud (occ maintenance:install) avec connexion RDS et S3.
Phase 3 Déploiement

Terraform Apply

  • Initialisation (terraform init).
  • Planification et vérification (terraform plan).
  • Application et récupération de l'URL Nextcloud (terraform apply).

Extraits de code clés

1. Réseau et sécurité IAM (main.tf)

Chaque ressource AWS est déclarée en code HCL. Le rôle IAM permet à l'EC2 d'accéder à S3 de façon sécurisée, sans stocker de clés d'accès. 

# Rôle IAM : seul EC2 peut l'assumer
resource "aws_iam_role" "ec2_s3_role" {
  assume_role_policy = jsonencode({
    Statement = [{
      Action    = "sts:AssumeRole"
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
    }]
  })
}

# Politique : accès S3 limité au bucket du projet
resource "aws_iam_role_policy" "ec2_s3_policy" {
  policy = jsonencode({
    Statement = [{
      Effect = "Allow"
      Action = ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"]
      Resource = [
        aws_s3_bucket.nextcloud_storage.arn,
        "${aws_s3_bucket.nextcloud_storage.arn}/*"
      ]
    }]
  })
}

2. Configuration automatique au démarrage (userdata.sh)

Ce script est injecté dans l'EC2 via Terraform et s'exécute une seule fois au premier démarrage. Il installe et configure Nextcloud sans aucune intervention humaine. 

# Installation de la stack complète
dnf install -y httpd php8.4 php8.4-mysqlnd php8.4-gd   php8.4-xml php8.4-mbstring php8.4-zip mariadb105

# Initialisation silencieuse de Nextcloud (connexion RDS + S3)
sudo -u apache php occ maintenance:install   --database      "mysql"   --database-host "${db_endpoint}"   --database-name "${db_name}"   --database-user "${db_user}"   --database-pass "${db_password}"   --admin-user    "admin"   --admin-pass    "NextcloudAdmin123!"

# Connexion S3 via le rôle IAM de l'instance (sans clé en dur)
sudo -u apache php occ config:system:set   objectstore arguments use_instance_credentials   --value=true --type=boolean

🔧 Difficultés rencontrées

RDS : 2 sous-réseaux obligatoires

Problème : AWS RDS exige au minimum 2 sous-réseaux dans 2 zones de disponibilité différentes pour créer un DB Subnet Group.

Solution : Création d'un second subnet privé dans une AZ distincte (names[1]), regroupés dans un DB Subnet Group dédié.

Accès S3 sans clé AWS

Problème : Stocker des clés AWS dans le script est une faille de sécurité majeure (exposition dans les logs, GitHub…).

Solution : Rôle IAM attaché à l'instance EC2. Nextcloud utilise automatiquement les credentials de l'instance via le metadata service AWS.

Compatibilité PHP 8.4

Problème : Certaines extensions PHP nécessaires à Nextcloud n'étaient pas disponibles dans les dépôts par défaut d'Amazon Linux 2023.

Solution : Identification des paquets corrects (php8.4-*) et ajustement du script d'installation.

✅ Résultats

L'infrastructure complète se déploie en une seule commande (terraform apply) en environ 10 minutes, sans aucune intervention manuelle. Nextcloud est accessible via l'IP publique de l'EC2 dès la fin du déploiement, avec les fichiers stockés sur S3 et la base de données gérée par RDS. 

# Outputs affichés automatiquement après terraform apply
nextcloud_url    = "http://<ip-publique>"
ssh_connection   = "ssh -i ~/.ssh/nextcloud-key.pem ec2-user@<ip-publique>"
rds_endpoint     = "<endpoint>.rds.amazonaws.com"
s3_bucket_name   = "nextcloud-sio-storage"

🚀 Perspectives et Améliorations

  • Ajouter un certificat SSL/TLS (HTTPS) via AWS Certificate Manager.
  • Mettre en place un Load Balancer pour la haute disponibilité.
  • Stocker le tfstate dans S3 pour permettre le travail en équipe sans conflits.
  • Utiliser Ansible en complément pour une gestion de configuration plus fine.
