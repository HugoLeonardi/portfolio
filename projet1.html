Projet 1

Déploiement Automatisé Nextcloud sur AWS

Infrastructure as Code (IaC) avec Terraform — Architecture trois tiers

🎯 Contexte et Problématique

Situation initiale

Dans le cadre de ma mission E6, j'ai eu à répondre à un besoin de stockage collaboratif sécurisé pour une entreprise. La solution devait permettre le partage de fichiers entre collaborateurs, sans dépendre d'un serveur physique sur site (on-premise), tout en garantissant la confidentialité des données. Le déploiement devait être entièrement automatisé et reproductible.

Problématique

Comment déployer une infrastructure Nextcloud complète et sécurisée sur AWS, en séparant les couches applicative, base de données et stockage, de façon reproductible et sans aucune intervention manuelle ?

🎯 Objectifs du Projet

1

Infrastructure as Code

Provisionner l'ensemble des ressources AWS (VPC, EC2, RDS, S3, IAM) via Terraform de façon déclarative et versionnable.

2

Architecture trois tiers

Séparer les couches web (EC2), base de données (RDS MySQL) et stockage fichiers (S3) pour la sécurité et la scalabilité.

3

Sécurité Cloud

Appliquer le principe de moindre privilège via les Security Groups (filtrage réseau) et les rôles IAM (accès S3 sans clé en dur).

4

Automatisation complète

L'installation de Nextcloud et sa configuration sont entièrement réalisées par un script Bash injecté via le mécanisme user_data d'AWS.

🔍 Choix de la Solution

Solutions envisagées

❌ Solution 1 — Serveur physique on-premise

  • Contrôle total du matériel
  • Pas de coût d'abonnement Cloud
  • Coût matériel initial élevé
  • Maintenance physique nécessaire
  • Pas de redondance native
  • Déploiement manuel, non reproductible

⚠️ Solution 2 — Cloud manuel (Console AWS)

  • Accès à l'infrastructure AWS
  • Pas d'outillage IaC requis
  • Déploiement non reproductible
  • Risque d'erreur humaine élevé
  • Impossible à versionner
  • Difficile à maintenir en équipe

✅ Solution retenue — Terraform + AWS

  • Infrastructure 100% reproductible
  • Versionnée avec Git
  • Déploiement en une seule commande
  • Conforme aux pratiques DevOps
  • Reprise rapide en cas d'incident
  • Séparation claire des couches (réseau, BDD, stockage)

💡 Justification du choix

L'approche IaC garantit que l'infrastructure est toujours déployée avec la même configuration validée. Elle correspond aux pratiques DevOps actuelles et permet une reprise rapide en cas d'incident. Terraform permet de décrire l'infrastructure comme du code, de la versionner sur Git et de la redéployer à l'identique en une seule commande : terraform apply.

🏗️ Architecture Technique

L'infrastructure repose sur une architecture trois tiers déployée dans un VPC dédié, avec une séparation stricte des couches réseau, applicative, base de données et stockage.

Architecture trois tiers — VPC AWS avec EC2, RDS MySQL et S3

🌐
Couche Réseau — VPC et sous-réseaux

Le VPC 10.0.0.0/16 est découpé en trois sous-réseaux aux rôles distincts :

🟢 Sous-réseau public — 10.0.1.0/24

  • Héberge l'instance EC2 (Nextcloud)
  • Exposée sur les ports 80 (HTTP) et 22 (SSH)
  • Reliée à un Internet Gateway

🔴 Sous-réseau privé 1 — 10.0.2.0/24

  • Héberge le premier nœud RDS MySQL
  • Aucune route vers Internet
  • Totalement isolé du trafic public

🔴 Sous-réseau privé 2 — 10.0.3.0/24

  • Second nœud RDS dans une AZ différente
  • Requis par AWS pour la haute disponibilité
  • Aucune route vers Internet
⚙️
Couche Applicative — EC2 + Nextcloud

Une instance EC2 t2.micro (Ubuntu 22.04) héberge le serveur Apache, PHP et Nextcloud. La configuration initiale est entièrement automatisée via un script userdata.sh exécuté au premier démarrage.

🗄️
Couche Base de données — RDS MySQL

Une instance RDS MySQL 8.0 placée en sous-réseau privé stocke les métadonnées Nextcloud. Elle n'est accessible que depuis l'instance EC2, via un Security Group dédié autorisant uniquement le port 3306.

🪣
Couche Stockage — S3 + IAM

Les fichiers des utilisateurs sont stockés dans un bucket S3. L'accès se fait sans clé AWS exposée : un rôle IAM est attaché à l'instance EC2 via un Instance Profile, permettant à Nextcloud de récupérer des credentials temporaires automatiquement depuis le service de métadonnées AWS.

⚙️ Réalisation

Méthodologie

Le projet a été découpé en trois phases : écriture du code infrastructure (HCL Terraform), écriture du script de provisionnement (Bash), puis déploiement et tests.

Phase 1 Infrastructure

Codage Terraform

  • Définition du provider AWS (région eu-west-3 — Paris).
  • Création du VPC, des trois sous-réseaux, de l'Internet Gateway et des tables de routage.
  • Configuration des Security Groups EC2 et RDS.
  • Définition de l'instance EC2, du groupe de sous-réseaux RDS et de l'instance RDS MySQL.
  • Création du bucket S3 et du rôle IAM avec sa politique d'accès.
Phase 2 Provisionnement

Script userdata.sh

  • Installation automatique d'Apache, PHP et des extensions requises par Nextcloud.
  • Téléchargement et déploiement de Nextcloud dans le répertoire web Apache.
  • Configuration de la connexion à RDS MySQL (endpoint injecté par Terraform).
  • Configuration du stockage objet S3 via le rôle IAM de l'instance (sans clé en dur).
  • Configuration dynamique des trusted_domains avec l'IP publique récupérée depuis le service de métadonnées AWS (169.254.169.254).
Phase 3 Déploiement

Terraform Apply

  • Initialisation des providers (terraform init).
  • Planification et vérification des ressources à créer (terraform plan).
  • Déploiement complet de l'infrastructure (terraform apply).
  • Récupération automatique des outputs : URL Nextcloud, IP publique, endpoint RDS, nom du bucket S3.

Extraits de code significatifs

1. Rôle IAM — Accès S3 sans clé (main.tf)

Le rôle IAM est la pièce maîtresse de la sécurité du projet. L'EC2 assume ce rôle automatiquement au démarrage, ce qui lui permet d'accéder à S3 sans qu'aucune clé ne soit stockée sur le serveur. 

resource "aws_iam_role" "ec2_s3_role" {
  name = "${var.project_name}-ec2-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action    = "sts:AssumeRole"
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
    }]
  })
}

resource "aws_iam_role_policy" "ec2_s3_policy" {
  name = "${var.project_name}-s3-policy"
  role = aws_iam_role.ec2_s3_role.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Action = ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"]
      Resource = [
        aws_s3_bucket.nextcloud_storage.arn,
        "${aws_s3_bucket.nextcloud_storage.arn}/*"
      ]
    }]
  })
}

# Lien entre le rôle et l'instance EC2
resource "aws_iam_instance_profile" "ec2_profile" {
  name = "${var.project_name}-ec2-profile"
  role = aws_iam_role.ec2_s3_role.name
}

2. Configuration du stockage S3 dans Nextcloud (userdata.sh)

Nextcloud est configuré pour utiliser S3 comme backend de stockage. Le paramètre use_instance_credentials = true indique à Nextcloud d'utiliser le rôle IAM de l'instance plutôt que des clés AWS statiques. 

sudo -u apache php occ config:system:set \
  objectstore arguments use_instance_credentials \
  --value=true \
  --type=boolean

3. Configuration des trusted_domains dynamiques (userdata.sh)

L'IP publique de l'EC2 change à chaque recréation de l'instance. Ce script récupère l'IP dynamiquement via le service de métadonnées AWS et la déclare comme domaine de confiance dans Nextcloud. 

sudo -u apache php /var/www/html/nextcloud/occ config:system:set \
  trusted_domains 1 \
  --value="$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4)"

4. Outputs Terraform — Informations post-déploiement (outputs.tf)

À la fin du terraform apply, ces outputs affichent directement dans le terminal toutes les informations nécessaires pour accéder à l'infrastructure. 

output "nextcloud_url" {
  value = "http://${aws_instance.nextcloud.public_ip}"
}

output "ssh_connection" {
  value = "ssh -i ~/.ssh/${var.key_pair_name}.pem ec2-user@${aws_instance.nextcloud.public_ip}"
}

output "rds_endpoint" {
  value = aws_db_instance.nextcloud.address
}

output "s3_bucket_name" {
  value = aws_s3_bucket.nextcloud_storage.bucket
}

Difficultés rencontrées et solutions

Difficulté : Les deux zones de disponibilité RDS

Problème : AWS RDS exige un "DB Subnet Group" contenant des sous-réseaux dans au moins deux zones de disponibilité (AZ) différentes. Avec un seul sous-réseau privé, Terraform renvoyait une erreur au déploiement.

Solution : Ajout d'un second sous-réseau privé (10.0.3.0/24) dans une deuxième AZ, déclaré dans le DB Subnet Group. Le data source aws_availability_zones permet de récupérer les AZ disponibles dynamiquement.

Difficulté : Accès S3 sans exposer de clés AWS

Problème : Nextcloud nécessite des credentials AWS pour accéder à S3. Stocker une Access Key directement dans le script Bash représente un risque de sécurité majeur (fuite via GitHub, logs, etc.).

Solution : Utilisation d'un rôle IAM attaché à l'instance EC2 via un Instance Profile. Nextcloud est configuré avec use_instance_credentials = true pour récupérer les credentials temporaires automatiquement depuis le service de métadonnées AWS.

Difficulté : IP publique dynamique et trusted_domains

Problème : Nextcloud bloque les accès depuis des domaines non déclarés dans sa configuration trusted_domains. Or l'IP publique de l'EC2 change à chaque terraform apply.

Solution : Le script userdata.sh interroge le service de métadonnées EC2 (169.254.169.254) pour récupérer l'IP publique au moment du démarrage de l'instance et la déclarer dynamiquement dans la configuration Nextcloud.

✅ Bilan

Le projet remplit l'ensemble des objectifs fixés : l'infrastructure Nextcloud est déployée intégralement par le code, en une seule commande terraform apply. L'architecture trois tiers garantit l'isolation des composants : la base de données n'est jamais exposée à Internet, et les fichiers utilisateurs sont persistants sur S3, indépendamment du cycle de vie de l'instance EC2.

Sur le plan de la sécurité, le principe de moindre privilège est appliqué à deux niveaux : les Security Groups limitent les flux réseau au strict nécessaire, et le rôle IAM restreint l'accès S3 au seul bucket du projet, sans aucune clé stockée en clair.

🚀 Perspectives et Améliorations

Évolutions possibles

  • Remote state Terraform : Stocker le fichier d'état terraform.tfstate dans un bucket S3 avec verrouillage DynamoDB pour permettre le travail en équipe sans conflits.
  • HTTPS avec ACM : Ajouter un certificat SSL/TLS via AWS Certificate Manager et un Load Balancer pour sécuriser les échanges entre les utilisateurs et Nextcloud.
  • AWS Secrets Manager : Externaliser les credentials de la base de données (mot de passe RDS) dans AWS Secrets Manager plutôt que dans les variables Terraform, pour renforcer la sécurité.
  • Ansible en complément : Remplacer le script Bash userdata.sh par un playbook Ansible, plus maintenable, idempotent et facilement modifiable sans recréer l'instance.

Ce que je referais différemment

Je mettrais en place le remote state dès le début du projet. Travailler avec un tfstate local est risqué : sa suppression accidentelle ferait perdre à Terraform la connaissance de l'état de l'infrastructure. Je configurerais également HTTPS dès le départ, car HTTP n'est pas acceptable pour une solution de stockage de fichiers en production.
