Projet 1

Déploiement Nextcloud sur AWS

Architecture trois tiers — EC2, RDS MySQL, S3

Contexte et Problématique

Situation initiale

Dans le cadre de ma formation BTS SIO SISR, j'ai eu à répondre à un besoin de stockage collaboratif sécurisé. La solution devait permettre le partage de fichiers entre collaborateurs sans dépendre d'un serveur physique sur site, tout en garantissant la confidentialité des données.

Problématique

Comment déployer une infrastructure Nextcloud complète et sécurisée sur AWS, en séparant les couches applicative, base de données et stockage, en maîtrisant chaque étape de configuration ?

Objectifs du Projet

1

Infrastructure Cloud

Provisionner les ressources AWS (VPC, EC2, RDS, S3, IAM) via la console AWS, en comprenant le rôle de chaque composant.

2

Architecture trois tiers

Séparer les couches web (EC2), base de données (RDS MySQL) et stockage fichiers (S3) pour la sécurité et la scalabilité.

3

Sécurité Cloud

Appliquer le principe de moindre privilège via les Security Groups (filtrage réseau) et les rôles IAM (accès S3 sans clé en dur).

4

Configuration serveur

Installer et configurer manuellement Nextcloud sur l'instance EC2 via SSH, en maîtrisant la pile Apache / PHP / MySQL.

Choix de la Solution

Solutions envisagées

❌ Solution 1 — Serveur physique on-premise

  • Contrôle total du matériel
  • Pas de coût d'abonnement Cloud
  • Coût matériel initial élevé
  • Maintenance physique nécessaire
  • Pas de redondance native
  • Scalabilité limitée

✅ Solution retenue — AWS via console

  • Infrastructure managée, haute disponibilité
  • Prise en main progressive de chaque service
  • Contrôle fin de chaque composant
  • Pas d'outil tiers requis
  • Services RDS et S3 natifs et fiables
  • Séparation claire des couches (réseau, BDD, stockage)

💡 Solution envisagée en évolution — Terraform

  • Infrastructure reproductible en une commande
  • Versionnée avec Git
  • Conforme aux pratiques DevOps
  • Courbe d'apprentissage plus élevée
  • Outillage supplémentaire requis

💡 Justification du choix

Le déploiement via la console AWS a été retenu pour maîtriser chaque composant de l'infrastructure de façon concrète. Cette approche permet de comprendre précisément le rôle de chaque ressource (VPC, Security Group, IAM) avant d'envisager leur automatisation. Terraform représente une évolution naturelle pour rendre cette infrastructure reproductible et versionnée.

Architecture Technique

L'infrastructure repose sur une architecture trois tiers déployée dans un VPC dédié, avec une séparation stricte des couches réseau, applicative, base de données et stockage.

Architecture trois tiers — VPC AWS avec EC2, RDS MySQL et S3

🌐
Couche Réseau — VPC et sous-réseaux

Le VPC 10.0.0.0/16 est découpé en trois sous-réseaux aux rôles distincts :

🟢 Sous-réseau public — 10.0.1.0/24

  • Héberge l'instance EC2 (Nextcloud)
  • Exposée sur les ports 80 (HTTP) et 22 (SSH)
  • Reliée à un Internet Gateway

🔴 Sous-réseau privé 1 — 10.0.2.0/24

  • Héberge le premier nœud RDS MySQL
  • Aucune route vers Internet
  • Totalement isolé du trafic public

🔴 Sous-réseau privé 2 — 10.0.3.0/24

  • Second nœud RDS dans une AZ différente
  • Requis par AWS pour la haute disponibilité
  • Aucune route vers Internet
⚙️
Couche Applicative — EC2 + Nextcloud

Une instance EC2 T3.small (Amazon Linux 2023) héberge le serveur Apache, PHP et Nextcloud. L'installation a été réalisée manuellement via SSH après le lancement de l'instance depuis la console AWS.

🗄️
Couche Base de données — RDS MySQL

Une instance RDS MySQL 8.0 placée en sous-réseau privé stocke les métadonnées Nextcloud. Elle n'est accessible que depuis l'instance EC2, via un Security Group dédié autorisant uniquement le port 3306.

🪣
Couche Stockage — S3 + IAM

Les fichiers des utilisateurs sont stockés dans un bucket S3. L'accès se fait sans clé AWS exposée : un rôle IAM est attaché à l'instance EC2 via un Instance Profile, permettant à Nextcloud d'accéder au bucket sans stocker de credentials sur le serveur.

Réalisation

Méthodologie

Le projet a été découpé en quatre phases : configuration du réseau, déploiement et configuration du serveur EC2, mise en place de la base de données RDS, puis configuration du stockage S3 et des droits IAM.

Phase 1 Réseau

Configuration du VPC (Console AWS)

  • Création du VPC 10.0.0.0/16 dans la région eu-west-3 (Paris).
  • Création des trois sous-réseaux : un public (10.0.1.0/24) et deux privés (10.0.2.0/24, 10.0.3.0/24) dans deux zones de disponibilité différentes.
  • Création et attachement d'un Internet Gateway au VPC.
  • Configuration de la table de routage du sous-réseau public vers l'Internet Gateway.
  • Création des Security Groups : un pour l'EC2 (ports 80 et 22), un pour le RDS (port 3306 depuis l'EC2 uniquement).
Phase 2 Serveur

Déploiement et configuration EC2

  • Lancement d'une instance EC2 T3.small (Amazon Linux 2023) dans le sous-réseau public.
  • Génération et association d'une paire de clés SSH pour l'accès sécurisé.
  • Connexion SSH à l'instance et installation d'Apache, PHP et des extensions requises par Nextcloud.
  • Téléchargement et déploiement de Nextcloud dans le répertoire web Apache.
  • Ajustement des permissions sur les répertoires Nextcloud (chown -R apache:apache).
Phase 3 Base de données

Configuration RDS MySQL

  • Création d'un DB Subnet Group associant les deux sous-réseaux privés.
  • Lancement d'une instance RDS MySQL 8.0 dans le sous-réseau privé, associée au Security Group RDS.
  • Création de la base de données et de l'utilisateur Nextcloud via connexion MySQL depuis l'EC2.
  • Configuration de la connexion dans config.php de Nextcloud (endpoint RDS, port 3306).
Phase 4 Stockage

S3 et IAM

  • Création d'un bucket S3 dédié au stockage des fichiers Nextcloud.
  • Création d'un rôle IAM autorisant l'accès S3 (GetObject, PutObject, DeleteObject, ListBucket) uniquement sur ce bucket.
  • Attachement du rôle IAM à l'instance EC2 via un Instance Profile (sans stocker de clé AWS sur le serveur).
  • Configuration du stockage objet dans Nextcloud avec use_instance_credentials = true.

Commandes clés exécutées sur l'EC2

1. Installation de la pile Apache / PHP

Après connexion SSH, installation des paquets nécessaires à Nextcloud sur Amazon Linux 2023. 

sudo dnf install -y httpd php php-mysqlnd php-gd php-xml \
  php-mbstring php-curl php-zip php-intl php-bcmath

sudo systemctl enable --now httpd

2. Configuration du stockage S3 dans Nextcloud

Nextcloud est configuré pour utiliser S3 comme backend de stockage. Le paramètre use_instance_credentials = true indique à Nextcloud d'utiliser le rôle IAM de l'instance plutôt que des clés AWS statiques. 

sudo -u apache php /var/www/html/nextcloud/occ config:system:set \
  objectstore arguments use_instance_credentials \
  --value=true \
  --type=boolean

3. Déclaration des trusted_domains

Nextcloud bloque les accès depuis des domaines non déclarés dans sa configuration. L'IP publique de l'EC2 est récupérée depuis le service de métadonnées AWS et déclarée comme domaine de confiance. 

sudo -u apache php /var/www/html/nextcloud/occ config:system:set \
  trusted_domains 1 \
  --value="$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4)"

Difficultés rencontrées et solutions

Difficulté : Les deux zones de disponibilité pour RDS

Problème : AWS RDS exige un "DB Subnet Group" contenant des sous-réseaux dans au moins deux zones de disponibilité différentes. Avec un seul sous-réseau privé, la création du RDS était bloquée.

Solution : Ajout d'un second sous-réseau privé (10.0.3.0/24) dans une deuxième AZ, puis création du DB Subnet Group en associant les deux sous-réseaux privés.

Difficulté : Accès S3 sans exposer de clés AWS

Problème : Nextcloud nécessite des credentials AWS pour accéder à S3. Stocker une Access Key directement dans la configuration représente un risque de sécurité majeur (fuite, etc.).

Solution : Utilisation d'un rôle IAM attaché à l'instance EC2 via un Instance Profile. Nextcloud est configuré avec use_instance_credentials = true pour récupérer les credentials temporaires automatiquement depuis le service de métadonnées AWS.

Difficulté : IP publique dynamique et trusted_domains

Problème : Nextcloud bloque les accès depuis des domaines non déclarés dans sa configuration trusted_domains. Après un redémarrage d'instance, l'IP publique peut changer.

Solution : Interrogation du service de métadonnées EC2 (169.254.169.254) pour récupérer l'IP publique courante et la déclarer dynamiquement dans la configuration Nextcloud.

Bilan

Le projet remplit l'ensemble des objectifs fixés : l'infrastructure Nextcloud est opérationnelle sur AWS avec une architecture trois tiers. La base de données n'est jamais exposée à Internet, et les fichiers utilisateurs sont persistants sur S3, indépendamment du cycle de vie de l'instance EC2.

Sur le plan de la sécurité, le principe de moindre privilège est appliqué à deux niveaux : les Security Groups limitent les flux réseau au strict nécessaire, et le rôle IAM restreint l'accès S3 au seul bucket du projet, sans aucune clé stockée en clair. Ce projet m'a permis de maîtriser concrètement les services AWS fondamentaux (VPC, EC2, RDS, S3, IAM) en configurant chacun d'eux manuellement.

Perspectives et Améliorations

Automatisation avec Terraform

L'évolution naturelle de ce projet serait d'automatiser l'ensemble de ces étapes via Terraform (Infrastructure as Code). Chaque ressource créée manuellement dans la console (VPC, Security Groups, EC2, RDS, S3, IAM) se traduirait en un bloc de code HCL déclaratif, versionnable sur Git et redéployable en une seule commande terraform apply. Cela éliminerait le risque d'erreur humaine et faciliterait la collaboration en équipe.

Autres évolutions possibles

  • HTTPS : Ajouter un certificat SSL/TLS pour sécuriser les échanges entre les utilisateurs et Nextcloud.
  • IP Élastique : Associer une Elastic IP à l'instance EC2 pour conserver la même adresse publique après chaque redémarrage, ce qui simplifierait la gestion des trusted_domains.
  • Supervision avec CloudWatch : Créer une alarme sur la métrique CPUUtilization de l'instance EC2 pour être alerté dès que la charge dépasse un seuil critique (ex : 80 %), et ainsi anticiper les problèmes de performance avant qu'ils n'impactent les utilisateurs.

Ce que je referais différemment

  • HTTPS dès le départ : HTTP n'est pas acceptable pour une solution de stockage de fichiers, même en lab. J'aurais configuré un certificat SSL dès l'installation.
  • Elastic IP : Associer une IP fixe à l'EC2 dès le début pour éviter de devoir mettre à jour les trusted_domains à chaque redémarrage.
  • Nom de domaine via Route 53 : Un enregistrement DNS personnalisé rend l'accès plus lisible et professionnel qu'une IP brute.
  • Secrets Manager : Externaliser les credentials de la base de données dans AWS Secrets Manager plutôt que dans le fichier de configuration, pour renforcer la sécurité.
