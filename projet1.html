📋 Contexte et Problématique

Dans le cadre de ma formation BTS SIO SISR, j'ai réalisé ce projet pour répondre à une problématique concrète rencontrée en entreprise : comment mettre en place une solution de stockage et de partage de fichiers (Nextcloud) sur le Cloud, de façon automatisée, sécurisée et reproductible ?

L'enjeu était d'aller au-delà d'un déploiement manuel classique et d'adopter une approche Infrastructure as Code (IaC) avec Terraform, qui est aujourd'hui le standard dans les équipes DevOps et les services informatiques. Chaque ressource AWS (réseau, serveur, base de données, stockage) est décrite dans des fichiers de code versionnables et réutilisables.

🎯 Problématique : Comment déployer une infrastructure Nextcloud fiable, sécurisée et reproductible sur AWS, en moins de 15 minutes, sans aucune intervention manuelle ?

🎯 Objectifs du Projet

1

Infrastructure as Code

Décrire et provisionner l'intégralité de l'infrastructure AWS via Terraform (VPC, EC2, RDS, S3, IAM).

2

Architecture trois tiers

Séparer clairement les couches : serveur web (EC2), base de données (RDS MySQL) et stockage fichiers (S3).

3

Sécurité & Moindre Privilège

Utiliser les rôles IAM pour l'accès S3 (sans clé en dur), isoler la BDD en sous-réseau privé, filtrer les flux réseau.

4

Automatisation complète

Installer et configurer Nextcloud automatiquement via un script User Data au premier démarrage du serveur.

🏗️ Architecture Technique

L'architecture repose sur le principe de séparation des couches. Chaque service a une responsabilité unique, ce qui facilite la maintenance, la scalabilité et la sécurité.

Schéma de l'infrastructure AWS — Architecture trois tiers Nextcloud

🌐

Couche Réseau — VPC & Security Groups

Un VPC (Virtual Private Cloud) isole l'infrastructure du reste d'AWS sur le plan réseau (CIDR 10.0.0.0/16). Il contient plusieurs sous-réseaux :

  • Subnet public : héberge l'EC2, accessible depuis Internet (ports 80 et 22 uniquement).
  • Subnets privés (x2, dans 2 zones de disponibilité différentes) : hébergent RDS, inaccessibles depuis l'extérieur.

Le Security Group joue le rôle de pare-feu : il n'autorise que le HTTP (port 80) et le SSH (port 22) en entrée. Toute autre connexion est bloquée par défaut.

🖥️

Couche Application — EC2 (t3.micro)

L'instance EC2 t3.micro sous Amazon Linux 2023 héberge le serveur web Apache, PHP 8.2 et l'application Nextcloud. Elle est configurée automatiquement au premier démarrage via un script User Data (Bash).

Un rôle IAM est attaché à cette instance, lui permettant d'accéder au bucket S3 sans avoir besoin de stocker des clés AWS dans le code — ce qui est une bonne pratique de sécurité fondamentale.

🗄️

Couche Données — RDS MySQL 8.0

La base de données est hébergée sur AWS RDS, un service de base de données managé. Cela signifie qu'AWS gère automatiquement les sauvegardes, les patches de sécurité et la disponibilité.

RDS est placé dans les subnets privés : il est accessible uniquement par l'EC2 (via le port 3306), jamais depuis Internet. AWS exige au minimum 2 subnets dans 2 zones de disponibilité différentes pour créer un DB Subnet Group.

🪣

Couche Stockage — AWS S3

Les fichiers des utilisateurs Nextcloud sont stockés dans un bucket S3 plutôt que sur le disque local de l'EC2. Cette approche présente plusieurs avantages :

  • Stockage pratiquement illimité et hautement disponible.
  • Les fichiers survivent à la destruction/recréation du serveur EC2.
  • Coût optimisé : on ne paye que ce qu'on stocke réellement.

L'accès S3 est sécurisé via un rôle IAM avec des permissions minimales (GetObject, PutObject, DeleteObject, ListBucket sur ce bucket uniquement).

⚙️ Réalisation Technique

Organisation des fichiers Terraform

Le projet est structuré en plusieurs fichiers Terraform, chacun ayant un rôle précis. Cette organisation est une bonne pratique qui facilite la lisibilité et la maintenance.

📄
main.tf

Ressources principales : VPC, subnets, EC2, RDS, S3, IAM. Cœur de l'infrastructure.

📄
variables.tf

Déclaration de toutes les variables (région, noms, CIDR...). Permet de personnaliser sans toucher au code principal.

📄
outputs.tf

Affiche les informations utiles après déploiement : IP publique, URL Nextcloud, endpoint RDS.

📄
userdata.sh

Script Bash exécuté automatiquement au 1er démarrage de l'EC2. Installe Apache, PHP, Nextcloud et configure tout.

Points techniques clés

1. Sécurité IAM — Accès S3 sans clé en dur

C'est l'un des points les plus importants du projet. Plutôt que de stocker des clés AWS dans la configuration de Nextcloud (mauvaise pratique, risque de fuite), un rôle IAM est attaché directement à l'instance EC2. Nextcloud utilise automatiquement les credentials de l'instance sans qu'aucune clé n'apparaisse dans le code.

main.tf — Rôle IAM et politique S3
# Rôle IAM : seul EC2 peut l'assumer
resource "aws_iam_role" "ec2_s3_role" {
  name = "${var.project_name}-ec2-role"
  assume_role_policy = jsonencode({
    Statement = [{
      Action    = "sts:AssumeRole"
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
    }]
  })
}

# Politique : accès limité à CE bucket uniquement
resource "aws_iam_role_policy" "ec2_s3_policy" {
  policy = jsonencode({
    Statement = [{
      Effect = "Allow"
      Action = ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"]
      Resource = [
        aws_s3_bucket.nextcloud_storage.arn,
        "${aws_s3_bucket.nextcloud_storage.arn}/*"
      ]
    }]
  })
}

2. Isolation réseau — RDS en sous-réseau privé

La base de données ne doit jamais être accessible depuis Internet. Elle est placée dans des subnets privés et un Security Group dédié n'autorise les connexions MySQL (port 3306) qu'en provenance de l'EC2. AWS oblige à créer un DB Subnet Group avec au minimum 2 sous-réseaux dans 2 zones de disponibilité.

main.tf — DB Subnet Group et Security Group RDS
# Groupe de subnets pour RDS (2 AZ minimum obligatoires)
resource "aws_db_subnet_group" "nextcloud" {
  subnet_ids = [aws_subnet.private_1.id, aws_subnet.private_2.id]
}

# Security Group RDS : port 3306 uniquement depuis l'EC2
resource "aws_security_group" "rds_sg" {
  ingress {
    from_port       = 3306
    to_port         = 3306
    protocol        = "tcp"
    security_groups = [aws_security_group.ec2_sg.id]  # EC2 seulement !
  }
}

3. Configuration automatique — User Data

Le script userdata.sh est injecté dans l'instance EC2 via Terraform et s'exécute automatiquement au premier démarrage. Il installe la stack complète et configure Nextcloud sans aucune intervention manuelle, y compris les trusted_domains avec l'IP publique réelle.

userdata.sh — Extraits clés
#!/bin/bash
# Installation de la stack (Apache, PHP 8.2, extensions Nextcloud)
dnf install -y httpd php php-mysqlnd php-gd php-xml php-mbstring php-zip ...

# Téléchargement et installation de Nextcloud
wget https://download.nextcloud.com/server/releases/nextcloud-27.1.5.zip
unzip nextcloud-27.1.5.zip -d /var/www/html/
chown -R apache:apache /var/www/html/nextcloud

# Configuration du stockage S3 (via rôle IAM, pas de clé !)
sudo -u apache php occ config:system:set \
  objectstore arguments use_instance_credentials \
  --value=true --type=boolean

# Trusted domains : ajout automatique de l'IP publique réelle
PUBLIC_IP=$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4)
sudo -u apache php /var/www/html/nextcloud/occ config:system:set \
  trusted_domains 1 --value="$PUBLIC_IP"

🔧 Difficultés Rencontrées et Solutions

🔴 RDS : 2 subnets dans 2 zones de disponibilité obligatoires

Problème : Lors de la création du DB Subnet Group pour RDS, Terraform retournait une erreur car AWS exige obligatoirement des subnets dans au moins 2 zones de disponibilité différentes (ex: us-east-1a et us-east-1b).

Solution : Création d'un second subnet privé dans une AZ distincte, puis regroupement des deux dans le DB Subnet Group. L'utilisation du data source aws_availability_zones permet de récupérer dynamiquement les AZ disponibles sans les coder en dur.

🔴 Erreur trusted_domains au premier accès

Problème : Nextcloud refusait toutes les connexions avec le message "Veuillez contacter votre administrateur" car l'IP publique de l'EC2 n'était pas dans la liste des trusted_domains.

Solution : Ajout dans le script User Data de commandes occ qui récupèrent automatiquement l'IP publique via l'API de métadonnées AWS (169.254.169.254) et la définissent comme trusted domain au moment de l'installation.

🔴 Compatibilité PHP 8.4 avec Nextcloud

Problème : La version de PHP installée par défaut sur Amazon Linux 2023 était trop récente et non supportée par Nextcloud 27, causant des erreurs à l'installation.

Solution : Spécification explicite de PHP 8.2 dans le script User Data et vérification de la compatibilité dans la documentation officielle Nextcloud avant le déploiement.

✅ Bilan et Perspectives

Ce que ce projet m'a apporté

Ce projet m'a permis de maîtriser la logique d'une architecture Cloud multi-services, d'appréhender concrètement les enjeux de sécurité réseau (isolation en subnet privé, IAM) et de comprendre pourquoi l'Infrastructure as Code est aujourd'hui incontournable dans les équipes informatiques : tout est versionnable, reproductible et auditable.

Évolutions possibles

  • HTTPS : Ajouter un certificat SSL via AWS Certificate Manager et un Load Balancer pour chiffrer les échanges.
  • Haute disponibilité : Activer le mode Multi-AZ sur RDS pour basculement automatique en cas de panne.
  • Backend Terraform distant : Stocker le fichier tfstate dans S3 pour permettre le travail en équipe sans conflit.
  • Ansible : Compléter Terraform avec Ansible pour la configuration fine de Nextcloud (gestion des utilisateurs, plugins...).

Ce que je referais différemment

J'aurais ajouté HTTPS dès le départ plutôt qu'en amélioration. En production, exposer une application de partage de fichiers en HTTP non chiffré n'est pas acceptable. J'aurais également utilisé AWS Secrets Manager pour stocker les mots de passe de la base de données plutôt que de les passer en variables Terraform.
